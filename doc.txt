19. Песочница SandboxApp для безопасных экспериментов.

КОМПОНЕНТЫ СИСТЕМЫ

АГЕНТЫ (4):
- OrchestratorAgent — главный оркестратор. Принимает запрос, получает контекст от ContextAgent, строит системный промпт с {TOOLS}, вызывает LLM, парсит JSON-ответы (ExtractActionPlanRobust), вызывает инструменты через ToolRegistry.ExecuteToolAsync. Цикл: Explore -> Read -> Write -> Validate -> Document -> Report. Защита от зацикливания (MaxIterations=10). Инжектит: ToolRegistry, ILlmService, ContextAgent, BuildValidationService, ReportService, MetricsService. Выбирает модель (SelectModel) и промпт по ключевым словам: simple (7B), default/bugfix/feature (14B), refactoring (16B).
- CodeEditorAgent — чтение/изменение файлов. Инструменты: ReadFileTool, WriteFileTool.
- DocumentationAgent — работа с документацией. Инструменты: ReadDocumentationTool, UpdateDocumentationTool.
- ContextAgent — управление контекстом (ProjectContext). Методы: SetProjectPath, CacheProjectStructure, RegisterFileModification, RegisterError, RecordInteraction, SetPhase, GetContextSummary, ResetErrors.

ИНСТРУМЕНТЫ (9, все реализуют IAgentTool, регистрируются через [AgentTool] и AddAgentTools):
- read_file — чтение файла. Зависимости: IOptions<AgentOptions>.
- write_file — запись файла с созданием директорий. Зависимости: IOptions<AgentOptions>.
- project_structure — сканирование структуры. Зависимости: IOptions<AgentOptions>.
- read_documentation — семантический поиск по Qdrant. Зависимости: IOllamaEmbeddingClient, IQdrantService, IOptions<QdrantOptions>.
- update_documentation — обновление/создание секций с реиндексацией. Зависимости: IDocumentationService, IOllamaEmbeddingClient, IQdrantService, IOptions<AgentOptions>, IOptions<QdrantOptions>.
- run_shell_command — выполнение команд (белый список: dotnet, git, npm, docker, echo, dir, ls, cat, type). Зависимости: CliWrap, IOptions<AgentOptions>.
- git_diff — Git status/diff/log. Зависимости: LibGit2Sharp, IOptions<AgentOptions>.
- search_codebase — семантический поиск по коду через Qdrant (коллекция project_docs_code). Зависимости: IOllamaEmbeddingClient, IQdrantService, IOptions<QdrantOptions>.
- find_usages — поиск использований класса/метода во всех .cs файлах. Зависимости: IOptions<AgentOptions>.

ПРОМПТЫ (6):

- code-editor.txt — редактирование кода (C# 12, XML-документация)

ТИПИЧНЫЙ WORKFLOW ОРКЕСТРАЦИИ

1. Пользователь отправляет запрос: "Добавь метод Multiply в Calculator".
2. ContextAgent предоставляет сводку контекста.
3. Оркестратор начинает фазу планирования: search_codebase -> find_usages -> read_file.
4. LLM формирует план (action: "plan") со списком шагов.
5. ChangePlan наполняется шагами, выполняется с отслеживанием прогресса.
6. Каждый шаг: LLM возвращает delegate с нужным инструментом.
7. ExtractActionPlanRobust парсит JSON, ExecuteToolFromPlan вызывает инструмент.
8. Результат добавляется в conversationHistory (StringBuilder).
9. Цикл повторяется: read_file -> write_file -> run_shell_command (dotnet build).
10. BuildValidationService выполняет сборку, при ошибке — до 3 ретраев.
11. При успехе: git_diff -> report.
12. ContextAgent записывает взаимодействие в историю.
13. MetricsService записывает метрики запроса.
14. ReportService генерирует форматированный отчёт.

ВАЖНЫЕ ЗАМЕТКИ

- Модель qwen2.5-coder:7b-instruct требует ~8 ГБ RAM. Можно заменить в appsettings.json.
- Qdrant: вектор 768 (embedding_length nomic-embed-text), Cosine distance.
- EmbeddingModel: nomic-embed-text. qwen2.5-coder не поддерживает embeddings capability.
- Ollama должна быть запущена с OLLAMA_EMBEDDINGS=1 в docker-compose.yml.
- Поиск по коду: коллекция project_docs_code (отдельно от project_docs).
- При выборе модели: первая итерация — модель по задаче, остальные — default (14B).
- FindUsagesTool исключает bin/obj/.git директории при поиске.
- Semantic Kernel используется только для Memory.
- Microsoft.Agents.Hosting исключён — собственная оркестрация.
- Dockerfile в корне проекта (рядом с .sln), не в ProjectAIAgent.Host/.
- Для CI/CD нужны секреты DOCKER_USERNAME и DOCKER_PASSWORD в GitHub.
- Все изменения кода через Git (LibGit2Sharp).
- Результаты read_file обрезаются до 3000 символов (защита контекстного окна).
- RunShellCommandTool: белый список команд, запрет rm/del/format.
- DocumentationAgent в тестовом режиме честно отказывается документировать без кода.
- PromptTesterService: /role для смены роли, /system для просмотра промпта.
- FileSystemWatcher: дебаунс 3 секунды, отключение через WatchDocumentation: false.
- MetricsService: метрики доступны через API (status) и CLI (status).
- Все инструменты регистрируются автоматически — достаточно добавить класс с [AgentTool] и IAgentTool.
- Специализированные промпты выбираются автоматически по ключевым словам в запросе пользователя.
- refactoring.txt используется для запросов с "рефакторинг", "перепиши", "выдели интерфейс".
- bugfix.txt для "исправь", "почини", "ошибка", "баг".
- feature.txt для "добавь", "создай", "реализуй".
- ExtractFailedFiles извлекает файлы с ошибками из вывода dotnet build для целевой диагностики.
- Расширение VS Code находится в папке vscode-extension/. Требует Node.js и vsce для упаковки.
- Горячая клавиша Ctrl+Shift+A отправляет выделенный код или запрос агенту.
- ApiUrl настраивается в settings.json: "aiagent.apiUrl": "http://localhost:5000".
